# DataWindow 灵魂永续 — ngx-datawindow 设计宣言

> PowerBuilder 几经易手，但 DataWindow 的设计哲学不该随之湮灭。
>
> 这个项目存在的意义，不是"造一个 Angular 表格组件"，而是**把 DataWindow 的思想在 Web 时代延续**。

---

## 一、为什么 DataWindow 值得被铭记

### 1.1 它回答了一个永恒的问题

> **"数据在 UI 和存储之间，应该如何被管理？"**

在 DataWindow 诞生之前（1991 年），这个问题的答案是：

- 发 SQL 查询数据
- 把数据填入 UI 控件
- 用户编辑后，再写 SQL 更新数据
- 校验逻辑散落在各处

DataWindow 的答案是：

- **数据由一个专门的引擎管理**
- UI 只是引擎的"视图"
- 引擎负责校验、状态跟踪、变更管理、持久化
- 开发者配置，而不是写重复代码

这个答案，在 1991 年是革命性的，在 2024 年依然是正确的。

### 1.2 它证明了一件事

> **"好的架构比多的功能更长久。"**

DataWindow 30 年不衰，不是因为功能多（虽然确实多），而是因为：

- 三缓冲区设计，让数据操作可逆
- 列级变更跟踪，让更新精准
- 事件链设计，让每个环节可干预
- 声明式配置，消灭重复代码

这些是**架构层面的正确决策**，不依赖具体技术栈，不随时间过时。

### 1.3 它的衰落是商业原因，不是技术原因

PowerBuilder 经 Sybase、SAP、Appeon 多次转手，市场推广不力，生态萎缩。DataWindow 本身的设计没有过时——它只是需要一个**新的载体**。

## 二、ngx-datawindow 的使命

### 2.1 不是"替代"，是"传承"

我们不追求 100% 功能对标，那没有意义。我们追求的是**继承 DataWindow 的设计灵魂**。

| DataWindow 的灵魂 | ngx-datawindow 的传承 |
|------------------|---------------------|
| 数据中间层哲学 | DataStore 引擎独立于组件 |
| 三缓冲区 + 可逆操作 | main / filtered / deleted 缓冲区 + Undo/Redo |
| 列级变更跟踪 | ✅ 已实现（Phase 1） |
| ItemChanged 拒绝机制 | ✅ 已实现（Phase 1） |
| 声明式配置 | ColumnConfig + TableConfig |
| 事件链 + 可干预 | ✅ 已实现（Phase 1） |
| 计算列一等公民 | 虚拟计算列 + JS 函数公式 |

### 2.2 在 Web 时代找到新价值

DataWindow 是 Windows C/S 时代的产物，有些东西注定无法直接移植。但有些东西在 Web 时代反而**更有价值**：

- **响应式 UI** — DataWindow 做不了的
- **跨平台** — DataWindow 只能 Windows
- **组件复用** — DataWindow 只能在 PowerBuilder 里用
- **JS 函数公式** — 比 PowerBuilder 表达式更强大
- **实时协作** — DataWindow 没有这个概念

**ngx-datawindow 的差异化价值** = DataWindow 的设计灵魂 + Web 原生优势

## 三、设计原则（从 DataWindow 继承）

### 原则一：数据是第一公民

> 数据不应该散落在 UI 控件里。数据有专门的引擎管理，UI 只是视图。

**实践：**

- DataStore 独立于 Angular 组件
- 组件可以销毁重建，DataStore 可以保留
- 同一 DataStore 可以绑定多个视图（Grid / Form / Chart）

### 原则二：操作是可逆的

> 用户的操作不应该立即生效，应该有"暂存 → 确认 → 生效"的过程。

**实践：**

- 删除不是 splice，是移到 delete 缓冲区
- 修改不是直接赋值，是记录变更历史
- 提供撤销/回滚能力

### 原则三：变更是可追溯的

> 系统应该知道数据从哪来、变成啥、谁改的、什么时候改的。

**实践：**

- 每行有状态：new / normal / modified / deleted
- 每列有变更记录：oldValue / newValue + timestamp
- 生成差异更新时精确到列

### 原则四：校验是实时的

> 无效数据不应该进入缓冲区，应该在入口处被拦截。

**实践：**

- ItemChanged 事件在值变化时触发
- 返回 reject 时，值不进入缓冲区，光标停留
- 数据进缓冲区前已经是干净的

### 原则五：每个环节可干预

> 框架提供钩子，开发者决定要不要干预。不是黑盒自动执行。

**实践：**

- 完整的事件链：RetrieveStart → RowFocusChanged → ItemChanged → SaveStart
- 每个事件可以阻止、修改、记录日志
- 开发者对数据流有完全控制

## 四、实现路线图

### Phase 1：核心灵魂 ✅ 已完成

让数据引擎具备"认真对待数据"的能力：

- ✅ 三缓冲区管理
- ✅ 行级状态跟踪
- ✅ 列级变更跟踪
- ✅ ItemChanged + 拒绝机制
- ✅ 撤销/回滚栈
- ✅ 完整事件链

### Phase 2：工程可靠性 ✅ 已完成

让数据操作经得起生产环境考验：

- ✅ 离线支持（IndexedDB 持久化）
- ✅ 冲突检测（乐观锁）
- ✅ 批量操作优化
- ✅ 大数据量虚拟滚动
- ✅ 性能监控埋点

### Phase 3：开发体验（进行中）

让开发者用得爽：

- [ ] 可视化列配置设计器
- [ ] 声明式持久化配置（自动生成 API 调用）
- [ ] 多种呈现样式（Grid / Form / Card）
- [ ] 报表导出（PDF / Excel）
- [ ] 完整文档 + 示例
